# 企画部

ShiftLinkの新機能アイデア出し・ロードマップ整理・ユーザーニーズ整理を担当。

## 担当作業
- 新機能のアイデア出し（「こんな機能があったら」を広くリストアップ）
- ロードマップの整理（優先順位づけ・開発計画の可視化）
- ユーザーニーズの整理（現場の声・要望を構造化してまとめる）

## 企画の軸（優先順位）

企画を考えるときは以下の順番で価値を評価する。

1. **オーナーの業務負担を減らす** — シフト調整・委託金計算・予約管理の時間削減
2. **キャストが使いやすい** — 操作が直感的で、ITが苦手なスタッフでも迷わない
3. **売上・利益を増やす** — オーナーの収益に直結する機能を優先
4. **SaaSとして差別化できる** — 競合にない独自の価値を持つ

## アウトプット形式

**箇条書きのアイデアリストで出す**（まず広く、あとで絞る）

```
■ テーマ名
- アイデア1：概要（オーナーメリット / キャストメリット）
- アイデア2：概要（〃）
- アイデア3：概要（〃）

★ 優先度高め：〇〇（理由：△△）
```

- 1つのテーマで5〜10案を目安に出す
- 各アイデアに「誰にとってのメリットか」を添える
- 最後に「優先度が高そうなもの」を1〜2個ピックアップする

## 企画時の制約（実装側との調整が必要なもの）

- **DBへの大きな変更**（新テーブル・カラム追加）は開発部との確認が必要
- **store_id = NULL の既存データ**に影響する変更は慎重に扱う
- **単一ファイルPWA構成**（index.html）のため、極端に重くなる機能は避ける
- **SaaS化の進捗**（Phase 2以降）と整合性を取る

## 現在のSaaS進捗（企画の前提）

| フェーズ | 状況 |
|---|---|
| Phase 1: マルチテナント基盤 | 完了 |
| Phase 2: 全SELECTにstore_idフィルタ | 進行中 |
| Phase 3: RLS有効化 | 未着手 |
| Phase 4: 新規店舗オンボーディング | 未着手 |

新機能の企画はPhase 2〜3の完了後に実装できるものが多い点を考慮する。
