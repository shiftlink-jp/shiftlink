# テスト部

ShiftLinkの修正後の動作確認・バグレポート作成・リグレッションチェックを担当するQA部門。

## 担当作業
- 修正後の動作確認（バグ修正・新機能が正しく動くか検証）
- バグレポートの作成（発見した不具合を整理してまとめる）
- リグレッションチェック（変更によって他の機能が壊れていないか確認）

## 重点チェックポイント

### 1. 既存データへの影響（最重要）
- `store_id = NULL` の既存データが正しく表示・操作できるか
- 新機能追加後も既存のPINログインアプリが正常に動作するか
- DB変更後にデータが欠損・変形していないか

### 2. 店舗間のデータ分離（SaaS）
- SaaSモードで自店舗のデータのみ表示されるか
- 別の `store_id` のデータが混入していないか
- RLS（行レベルセキュリティ）が正しく機能しているか

### 3. スマホでの操作感
- タップ・スクロールが意図通りに動くか
- ボトムシートの開閉が正常か
- フォーム入力時にキーボードが隠れて操作できなくなっていないか
- 文字・ボタンのサイズがスマホで見やすいか

### 4. 決済・通知フロー
- Stripe決済フロー（チェックアウト→webhook→DB更新）が正常か
- Push通知が正しい相手（オーナー/キャスト）に届くか
- 他店舗に通知が飛んでいないか（`store_id` フィルタの確認）

## リグレッションチェックリスト（変更後に必ず確認）

- [ ] PINログイン（オーナー・キャスト）が正常にできる
- [ ] 本日ページの予約一覧が表示される
- [ ] 委託金入力シートが開いて保存できる
- [ ] シフト表の表示・編集ができる
- [ ] 顧客情報の参照・編集ができる
- [ ] 管理画面（コース・オプション・ルーム設定）が動く
- [ ] Push通知の購読が外れていない

## ⚠️ Supabase側の設定確認（コード外・要手動確認）

RLS（行レベルセキュリティ）はクライアント側の `withStoreFilter()` だけでなく、
**Supabase Dashboard で RLS ポリシーが有効になっているか**を確認すること。

確認方法:
1. Supabase Dashboard → Table Editor → 対象テーブル → RLS設定
2. `005_rls_store_isolation.sql` が本番DBに適用済みか確認
3. 別の `store_id` アカウントでログインして他店舗データが見えないことを確認

## 参考資料の場所
